# Deploy

Scripts for provisioning a VPS and deploying **Some Store** (Medusa + Next.js Storefront).

## Prerequisites

- Ubuntu 22.04 VPS with root SSH access
- Docker installed locally (for cross-compilation)
- SSH key configured for VPS access for deploy user

## Scripts

| Script | Description |
|--------|-------------|
| `setup-vps.sh` | Provisions a clean VPS: create deploy user, Node.js, PM2, PostgreSQL, Redis, Nginx, firewall, SSL |
| `deploy-server.sh` | Builds and deploys Medusa server + worker (PM2) |
| `deploy-storefront.sh` | Builds and deploys Next.js storefront (PM2, standalone mode) |
| `run-script.sh` | Compiles a local `server/src/scripts/*.ts`, uploads to VPS, and executes via `npx medusa exec` |

## Deployment Order

Every script requires an environment argument — `prod` or `stage` — which determines which env files directory is used (`prod_envs/` or `stage_envs/`).

```
1. ./deploy/setup-vps.sh prod                        # One-time VPS provisioning
2. ./deploy/deploy-server.sh prod                    # Deploy Medusa backend
3. ./deploy/run-script.sh prod seed                  # Seed database (drops and recreates!)
4. ./deploy/run-script.sh prod create-admin-user --email admin@example.com --password <password>  # Create admin user
5. ./deploy/deploy-storefront.sh prod                # Deploy Next.js storefront
```

For staging, replace `prod` with `stage`.

If a required env file is missing, the script prints an error and exits before making any changes.

## Environment Files

All env files live in `deploy/prod_envs/` (production) or `deploy/stage_envs/` (staging). Create them manually before running scripts. The directory is selected by the `prod` / `stage` argument passed to each script.

Derived values (DATABASE_URL, REDIS_URL, CORS, etc.) are **generated on the VPS** by deploy scripts from `.env.setup-vps`. Only secrets and unique settings are stored in per-service env files.

| File | Used by | Description |
|------|---------|-------------|
| `{prod,stage}_envs/.env.setup-vps` | All scripts | VPS connection, domains, DB/Redis credentials (source of derived values) |
| `{prod,stage}_envs/.env.deploy-server` | `deploy-server.sh` | Secrets only: JWT_SECRET, COOKIE_SECRET, RESEND_API_KEY |
| `{prod,stage}_envs/.env.deploy-worker` | `deploy-server.sh` | Secrets only: JWT_SECRET, COOKIE_SECRET, RESEND_API_KEY |
| `{prod,stage}_envs/.env.deploy-storefront` | `deploy-storefront.sh` | Secrets + unique: publishable key, revalidate secret, Stripe keys, S3 config |

### prod_envs/.env.setup-vps

```env
# IP address or hostname of the VPS
VPS_HOST=<your-vps-ip>

# SSH user for initial connection (usually root for setup-vps.sh)
VPS_SSH_USER=root

# Path to local SSH private key used for authentication deploy user
VPS_SSH_KEY=~/.ssh/id_ed25519

# Linux user created by setup-vps.sh for subsequent deploys
DEPLOY_USER=deploy

# Password for the deploy user (used for sudo)
DEPLOY_PASSWORD=<strong-password>

# Absolute path on the VPS where the project will be deployed
PROJECT_DIR=/var/www/some_store

# Node.js major version to install
NODE_VERSION=20

# PostgreSQL database name
DB_NAME=some_store

# PostgreSQL user with CREATEDB privilege
DB_USER=<db-user>

# PostgreSQL user password
DB_PASSWORD=<db-password>

# Redis password (set in redis.conf)
REDIS_PASSWORD=<redis-password>

# Domain for Medusa API (Nginx -> :9000)
API_DOMAIN=api.yourdomain.com

# Domain for Next.js storefront (Nginx -> :8000)
STORE_DOMAIN=yourdomain.com

# Domain for Medusa Admin dashboard (Nginx -> :9000)
ADMIN_DOMAIN=admin.yourdomain.com

# Email for SSL certificate registration (Certbot) and admin user
ADMIN_EMAIL=admin@yourdomain.com

# Whether to obtain SSL certificates via Certbot (true/false)
ENABLE_SSL=true
```

### prod_envs/.env.deploy-server

Secrets only — derived values (DATABASE_URL, CORS, etc.) are generated by `deploy-server.sh`:

```env
# Secret key for signing JWT tokens (generate with: openssl rand -hex 32)
JWT_SECRET=<random-hex-64>

# Secret key for encrypting cookies (generate with: openssl rand -hex 32)
COOKIE_SECRET=<random-hex-64>

# Resend API key for transactional emails (e.g. welcome, order confirmation)
RESEND_API_KEY=<resend-api-key>
```

### prod_envs/.env.deploy-worker

Same secrets as server:

```env
# Must match JWT_SECRET from .env.deploy-server
JWT_SECRET=<random-hex-64>

# Must match COOKIE_SECRET from .env.deploy-server
COOKIE_SECRET=<random-hex-64>

# Must match RESEND_API_KEY from .env.deploy-server
RESEND_API_KEY=<resend-api-key>
```

### prod_envs/.env.deploy-storefront

Unique settings + secrets — derived values (MEDUSA_BACKEND_URL, etc.) are generated by `deploy-storefront.sh`:

```env
# Medusa publishable API key (displayed after running: ./deploy/run-script.sh prod seed)
NEXT_PUBLIC_MEDUSA_PUBLISHABLE_KEY=pk_...

# Default region ISO-2 code for storefront pricing/checkout
NEXT_PUBLIC_DEFAULT_REGION=us

# Secret for protected revalidation endpoint calls
REVALIDATE_SECRET=<random-hex-64>

# ISR revalidation window in seconds (0 = always revalidate)
REVALIDATE_WINDOW=0

# Stripe publishable key (leave empty if Stripe is not enabled)
NEXT_PUBLIC_STRIPE_KEY=

# Medusa Payments publishable key (if Medusa Payments is used)
NEXT_PUBLIC_MEDUSA_PAYMENTS_PUBLISHABLE_KEY=

# Medusa Payments account ID (if Medusa Payments is used)
NEXT_PUBLIC_MEDUSA_PAYMENTS_ACCOUNT_ID=

# Optional S3 hostname for remote media assets
MEDUSA_CLOUD_S3_HOSTNAME=

# Optional S3 path/prefix for remote media assets
MEDUSA_CLOUD_S3_PATHNAME=
```

> The publishable API key is displayed after running `run-script.sh prod seed`.
> Add it to `prod_envs/.env.deploy-storefront`.

## PM2 Configuration

Each deploy script generates its own PM2 ecosystem file:

- `ecosystem.server.config.js` — medusa-server + medusa-worker
- `ecosystem.storefront.config.js` — storefront (port 8000)

Scripts use `pm2 delete` + `pm2 start` for clean restarts.

## Architecture

```
VPS (:9000)  ← Nginx ← api.yourdomain.com
  └─ medusa-server (PM2)
  └─ medusa-worker (PM2)

VPS (:8000)  ← Nginx ← yourdomain.com
  └─ storefront (PM2, Next.js standalone)

PostgreSQL (:5432) — some_store database
Redis (:6379) — session/cache store
```
